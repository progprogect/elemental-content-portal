# Contributing Guidelines

## Project Structure Rules

### Documentation
- **All documentation** must be placed in the `docs/` folder
- Do not add documentation files to the root directory
- Update this file if adding new documentation types

### Tests
- **All tests** should be placed in the `tests/` folder
- Use appropriate subdirectories for different test types (unit, integration, e2e)

### Code Organization
- Keep root directory clean
- Follow existing directory structure
- Use TypeScript for all new code

## Development Workflow

1. Create feature branch
2. Make changes
3. Test locally
4. Submit pull request

## Code Style

- Use TypeScript
- Follow ESLint rules
- Use Prettier for formatting
- Write clear, self-documenting code

## Commit Messages

- Use clear, descriptive messages
- Reference issue numbers if applicable

## Adding New Features

1. Update architecture documentation if needed
2. Add tests for new functionality
3. Update README if user-facing
4. Document API changes
